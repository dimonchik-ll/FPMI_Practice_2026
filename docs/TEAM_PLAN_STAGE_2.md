# План этапа 2 — параллельная разработка

## Цель этапа

Исправить визуальные проблемы башен и снарядов, добавить поворот врагов по дороге, создать большую карту для endless-режима и подготовить UI к выбору режима, паузе и финальным экранам.

Этот этап построен так, чтобы **четыре разработчика могли работать параллельно**. Каждый работает только в своей папке. Общая интеграция выполняется отдельным маленьким Pull Request после слияния четырёх независимых задач.

---

## 1. Один подготовительный commit делает тимлид

До создания рабочих веток тимлид создаёт отдельную ветку `chore/stage2-contracts` и меняет только:

```text
shared/contracts.py
```

### 1.1. Добавить действия UI

В `UiActionKind` добавить:

```python
TOGGLE_PAUSE = "toggle_pause"
SELECT_MODE = "select_mode"
SELECT_LEVEL = "select_level"
RESTART_GAME = "restart_game"
RETURN_TO_MENU = "return_to_menu"
```

### 1.2. Добавить состояние игры для UI

В `GameSnapshot` добавить поля с безопасными значениями по умолчанию:

```python
paused: bool = False
game_mode: str = "campaign"
selected_level: str = "campaign_1"
```

### 1.3. Зафиксировать правила отрисовки

Новые сигнатуры должны поддерживать старый запуск благодаря значениям по умолчанию:

```python
MapRenderer.draw(surface, game_map, camera=None)
TowerSystem.draw(surface, camera_offset=Vector2(0, 0))
EnemySystem.draw(surface, camera_offset=Vector2(0, 0))
```

После merge этого маленького PR файл `shared/contracts.py` снова замораживается. Никто из четырёх разработчиков больше его не меняет.

## 2. Владение кодом

| Разработчик | Ветка | Можно менять | Нельзя менять |
|---|---|---|---|
| 1. Карта и core | `dev-core-endless-map` | `core/`, `assets/maps/`, `assets/tiles/`, `tests/test_core_*.py` | `towers/`, `enemies/`, `ui/`, `shared/`, `integration/` |
| 2. Башни и снаряды | `dev-towers-projectiles` | `towers/`, `assets/tiles/towers/`, `assets/tiles/projectiles/`, `tests/test_towers_*.py` | `core/`, `enemies/`, `ui/`, `shared/`, `integration/` |
| 3. Враги и волны | `dev-enemies-direction-endless` | `enemies/`, `assets/tiles/enemies/`, `tests/test_enemies_*.py` | `core/`, `towers/`, `ui/`, `shared/`, `integration/` |
| 4. UI и экономика | `dev-ui-game-flow` | `ui/`, `assets/ui/`, `docs/balance.md`, `tests/test_ui_*.py` | `core/`, `towers/`, `enemies/`, `shared/`, `integration/` |

### Общие правила

- Нельзя импортировать реализацию соседнего модуля.
- Разрешены только типы из `shared/contracts.py`.
- Башни получают `tuple[EnemyView, ...]` и возвращают `list[DamageCommand]`.
- Враги принимают `list[DamageCommand]` и возвращают `list[GameEvent]`.
- UI получает `GameSnapshot` и возвращает только `UiAction`.
- Игровые координаты остаются мировыми. Камера влияет только на отрисовку и преобразование клика.

# Разработчик 1 — большая карта, камера и зоны строительства

## Цель

Сделать большую карту для endless-режима, добавить камеру и убрать постоянный build-маркер под уже построенной башней.

## Задачи

### 1. Убрать поле строительства под башней

Сейчас визуальный слой `Buildable` не должен рисоваться как постоянная часть карты.

Нужно:

- отображать маркер строительства только для свободной `BuildZone`;
- после `confirm_build()` сразу скрывать маркер;
- после `cancel_build()` или `release()` снова отображать маркер;
- не менять механику занятости клетки;
- не менять `BuildRequest` и маршрут.

**Ожидаемый результат:** до строительства виден маркер места под башню; после строительства остаётся обычная земля карты и сама башня, без дополнительного поля под ней.

### 2. Создать endless-карту

Создать файл:

```text
assets/maps/endless.tmx
```

Требования:

```text
минимум 60×38 клеток;
тайл 32×32;
маршрут минимум из 150 клеток дороги;
минимум 18 зон строительства;
один spawn и одна base на разных краях карты;
каждая зона строительства занимает 2×2 клетки.
```

В Tiled сохранить слои и объекты:

```text
Road
Objects
BuildZone
spawn
base
tower_spot
```

### 3. Создать камеру

Создать:

```text
core/camera.py
```

Класс `Camera` должен:

- хранить позицию камеры в мировых координатах;
- иметь `world_to_screen()`;
- иметь `screen_to_world()`;
- не позволять камере выйти за границы карты;
- двигаться стрелками и `W/A/S/D`;
- поддерживать перетаскивание средней кнопкой мыши.

### 4. Уровни

В `core/levels.py` добавить:

```text
campaign_1
campaign_2
endless
```

Не ломать старый API `GameMap` и `CoreWorld`.

### 5. Тесты

Добавить проверки:

- занятая BuildZone не рисует build-маркер;
- повторное строительство в занятой зоне запрещено;
- endless-карта имеет связный маршрут;
- endless-карта имеет минимум 18 BuildZone;
- камера не выходит за границы карты;
- преобразование `world → screen → world` сохраняет позицию.

## Готово, когда

```text
- Под построенной башней не видно Buildable-поля.
- Endless-карта загружается без ошибок.
- В окне фиксированного размера можно двигать камеру.
- core.sandbox запускается без towers/, enemies/ и ui/.
```

# Разработчик 2 — башни, чистые спрайты и правильные стрелы

## Цель

Сделать корректную отрисовку башен и снарядов, не смешивая это с логикой карты.

## Задачи

### 1. Проверить и очистить визуал башен

Нужно проверить `idle` и `attack` PNG всех трёх башен.

Нужно:

- не рисовать одновременно два спрайта одной башни;
- выровнять масштаб трёх башен;
- поставить точку опоры в центр зоны строительства 2×2;
- при необходимости обрезать прозрачные поля PNG по alpha-границам;
- не добавлять траву, Buildable-тайл или площадку в `TowerSystem.draw()`.

**Граница ответственности:** визуальный build-маркер убирает разработчик 1. Разработчик 2 отвечает только за то, чтобы сам спрайт башни не дублировал землю или посторонний фон.

### 2. Переделать снаряды в корректные стрелы

Проверить реальный layout `arrow_1.png` и извлекать именно спрайт стрелы, а не квадратный тайл из листа.

Требования:

- стрела создаётся у лука, не в центре башни;
- сохраняет соотношение сторон;
- поворачивается по направлению полёта;
- выглядит корректно при полёте вправо, влево, вверх и вниз;
- исчезает после попадания;
- обычная стрела исчезает, если цель умерла или ушла;
- пробивающая стрела продолжает движение по текущему направлению;
- splash-стрела наносит урон в радиусе попадания;
- вместо PNG не рисуется линия, если ассет найден.

Урон должен по-прежнему передаваться только через `DamageCommand`.

### 3. Роли трёх башен

Сохранить и довести до конца:

```text
Archer I — частая одиночная атака.
Archer II — пробивающая стрела.
Archer III — медленная стрела с уроном по области.
```

### 4. Уровни башен

Реализовать уровни `1 → 2 → 3`.

При повышении уровня меняются минимум два параметра:

```text
урон;
дальность;
скорость атаки;
скорость снаряда.
```

### 5. Поддержка камеры

Обновить только отрисовку:

```python
TowerSystem.draw(surface, camera_offset=Vector2(0, 0))
```

Игровые позиции башен не менять.

### 6. Тесты

- стрела создаётся у башни;
- урон появляется только при попадании;
- стрела удаляется при смерти цели;
- пробивающая стрела поражает несколько целей;
- splash-стрела создаёт несколько `DamageCommand`;
- поворот стрелы рассчитывается для четырёх направлений;
- camera_offset не меняет мировую позицию башни.

## Готово, когда

```text
- Башни не имеют лишней земли в собственном спрайте.
- Стрелы выглядят как стрелы и летят в правильную сторону.
- Урон создаётся только при попадании.
- towers.sandbox запускается без core/, enemies/ и ui/.
```

# Разработчик 3 — повороты врагов, анимации и endless-волны

## Цель

Сделать так, чтобы враги визуально шли по дороге и разворачивались на каждом повороте, а бесконечные волны продолжались после четвёртой.

## Задачи

### 1. Направление движения

Внутри `enemies/` создать локальный enum:

```text
Facing.UP
Facing.DOWN
Facing.LEFT
Facing.RIGHT
```

В runtime-состояние врага добавить:

```text
facing
last_move_direction
```

Перед движением к следующей точке маршрута:

- вычислять `target - position`;
- определять направление по главной оси;
- сохранять его в `facing`;
- менять спрайт после поворота дороги.

### 2. Направленные анимации

Для ходьбы выбирать:

```text
up_walk.png
down_walk.png
left_walk.png
right_walk.png
```

Правило fallback:

```text
1. Файл нужного направления.
2. Горизонтальный файл с pygame.transform.flip().
3. down_walk.png.
```

Не использовать `pygame.transform.rotate()` на 90 градусов для пиксельных персонажей.

### 3. Корректное движение по маршруту

Враг должен:

- идти по точкам route;
- менять направление на поворотах;
- не дрожать на клетке;
- не перескакивать через углы;
- сохранять последний facing при атаке базы;
- сохранять facing во время death-анимации.

### 4. Endless-волны

Первые четыре волны остаются кампанией.

Для волны 5 и дальше:

- увеличивать здоровье и количество врагов постепенно;
- не увеличивать скорость бесконечно;
- добавлять босса каждые три волны;
- ограничить минимальную задержку спавна;
- не создавать слишком много врагов одновременно.

`EnemySystem.start_wave(wave_number, route)` должен уметь запускать любое положительное значение wave_number.

### 5. Поддержка камеры

Обновить только отрисовку:

```python
EnemySystem.draw(surface, camera_offset=Vector2(0, 0))
```

Логика движения остаётся в мировых координатах.

### 6. Тесты

- враг смотрит вниз на участке вниз;
- враг смотрит вверх на участке вверх;
- враг меняет направление на углу;
- враг проходит маршрут с несколькими поворотами;
- fallback не падает без directional-файла;
- волны 5, 10 и 20 создаются;
- здоровье и численность растут, а скорость ограничена.

## Готово, когда

```text
- Враги визуально поворачиваются вместе с дорогой.
- Есть корректная walk-анимация направлений.
- Волны 5+ не ломают EnemySystem.
- enemies.sandbox запускается без core/, towers/ и ui/.
```

# Разработчик 4 — UI, пауза, меню и экономика

Сделать все экраны и действия UI независимо от реальной карты, камеры, башен и врагов.

## Задачи

```text
Кампания
Бесконечный режим

Endless map
Начать игру
```

UI возвращает только:

```text
SELECT_MODE
SELECT_LEVEL
```

### 2. Пауза и настройки

Добавить:

```text
Пауза
Продолжить
Настройки
Выйти в меню
```

Горячие клавиши:

```text
Escape — переключить паузу.
```

В настройках хранить и отображать:

```text
громкость музыки;

скорость игры ×1 / ×2.
```

Даже если звук ещё не подключён, UI должен хранить эти значения.

Отображать:

```text
деньги;
жизни;
счёт;

режим игры;
```


Создать:

```text

```

Описать:

```text
стартовые деньги;
стоимость башен;
награды за врагов;
```

- выбор режима создаёт `SELECT_MODE`;
- выбор уровня создаёт `SELECT_LEVEL`;
- Escape создаёт `TOGGLE_PAUSE`;
- недостаток денег блокирует покупку;
- UI работает на фейковом `GameSnapshot` без карты и врагов;
- экран конца игры отдаёт `RESTART_GAME` и `RETURN_TO_MENU`.

## Готово, когда

```text
- ui.sandbox запускается без core/, towers/ и enemies/.
- Все UI-экраны работают на GameSnapshot.
- UI не импортирует реализацию игровых модулей.
```

# 3. Интеграция после четырёх независимых Pull Request

Это отдельный небольшой PR, который делает тимлид после merge веток разработчиков 1–4.

Ветка:

```text
integration/stage2-game-flow
```

Можно менять:

```text
integration/app.py
tests/test_integration_*.py
```

Нужно:

1. Создать `Camera` и передать её в `MapRenderer`.
2. Передавать `camera_offset` в `EnemySystem.draw()` и `TowerSystem.draw()`.
3. Преобразовывать клик мыши через `camera.screen_to_world()` до строительства.
4. Рисовать UI без camera_offset.
5. В кампании показать victory после четвёртой волны.
6. В endless-режиме после каждой завершённой волны увеличивать wave_number, не включая victory.
7. Обрабатывать `TOGGLE_PAUSE`, `SELECT_MODE`, `SELECT_LEVEL`, `RESTART_GAME`, `RETURN_TO_MENU`.

## Финальная ручная проверка

```text
1. На свободном месте виден build-маркер.
2. После строительства build-маркер исчезает, остаётся только башня.
3. Стрела летит из лука в нужную сторону.
4. Враг меняет направление на каждом повороте дороги.
5. Камера движется, а правая UI-панель остаётся неподвижной.
6. В кампании игра заканчивается после четвёртой волны.
7. В endless волны продолжаются после четвёртой.
```

# 4. Git-порядок работы

## Сначала тимлид

```bash
git checkout main
git pull --ff-only
git checkout -b chore/stage2-contracts
# изменить shared/contracts.py
git add shared/contracts.py
git commit -m "feat: add stage 2 ui contracts"
git push -u origin chore/stage2-contracts
```

После Pull Request и merge в `main` каждый разработчик создаёт свою ветку от обновлённого `main`.

## Каждый разработчик

```bash
git checkout main
git pull --ff-only
git checkout -b <своя-ветка>
git push -u origin <своя-ветка>
```

Перед Pull Request:

```bash
python -m pytest -q
python -m <свой_модуль>.sandbox
git status
git add <свои_файлы>
git commit -m "feat: <краткое описание>"
git push
```

## Порядок слияния

```text
1. chore/stage2-contracts
2. dev-core-endless-map
3. dev-towers-projectiles
4. dev-enemies-direction-endless
5. dev-ui-game-flow
6. integration/stage2-game-flow
```

Первые четыре рабочие ветки не зависят друг от друга. Последняя ветка интеграции зависит только от того, что четыре PR уже merged.
